# 이중 게이트 컴팩션 — 요구사항 문서

> 작성일: 2026-04-04
> 상태: 리뷰 대기

---

## 1. 배경

현재 `ContextManager.add()`는 `_total_chars() > max_context_chars` 단일 조건으로 압축을 트리거한다.
이로 인해:
- **짧은 메시지 다수**: 문자 수는 낮지만 메시지 수가 많을 때 압축이 발생하지 않아 메시지 목록이 비대해짐
- **조기 압축**: 메시지 수가 적어도 단일 대용량 메시지(예: 긴 파일 읽기 결과)로 인해 불필요한 압축 트리거
- **참조 구현과의 불일치**: claw-code는 `compactable.len() > preserve_recent_messages AND estimated_tokens >= max_estimated_tokens` 이중 게이트를 사용

## 2. 목표

컴팩션 트리거를 이중 게이트로 변경하여:
1. 메시지 수와 토큰/문자 수 **양쪽 모두** 임계값을 초과할 때만 압축 실행
2. 불필요한 조기 압축 방지
3. 짧은 메시지 다수 축적 시에도 적절한 시점에 압축
4. 기존 `force_compress()` (사용자 `/compact` 명령)는 영향 없음

## 3. 기능 요구사항

### 3.1 이중 게이트 조건

압축은 다음 **두 조건이 동시에** 충족될 때만 트리거:

| 게이트 | 조건 | 기본 임계값 |
|--------|------|------------|
| 게이트 1 (메시지 수) | `len(self.messages) > keep_recent * 2` | `keep_recent * 2` (기본 8) |
| 게이트 2 (문자 수) | `_total_chars() > max_context_chars` | 기존 `max_context_chars` 유지 |

- 게이트 1만 충족 (메시지 많지만 크기 작음): 압축하지 않음
- 게이트 2만 충족 (메시지 적지만 크기 큼): 압축하지 않음
- 두 게이트 모두 충족: 압축 실행

### 3.2 게이트 1 임계값의 의미

`keep_recent * 2`는 `_compress()`가 보존하는 최근 메시지 수(`keep`)와 동일하다.
즉, 압축 대상 메시지(`old_msgs`)가 **실제로 존재할 때만** 압축을 시도한다는 의미이다.
이는 `_compress()` 내부의 기존 가드(`len(self.messages) <= keep`)와 일관성을 가진다.

### 3.3 force_compress 동작 유지

`force_compress()`는 이중 게이트의 영향을 받지 않는다:
- 기존과 동일하게 `len(self.messages) > keep_recent * 2` 조건만으로 실행
- 사용자의 명시적 `/compact` 명령은 항상 존중

### 3.4 실패 추적과의 상호작용

기존 압축 실패 시 `max_context_chars`를 1.3배씩 올리는 로직은 그대로 유지:
- 게이트 2의 임계값이 동적으로 변동하는 것은 기존 동작 그대로
- 게이트 1은 실패 추적의 영향을 받지 않음

## 4. 비기능 요구사항

### 4.1 성능

- `add()` 호출 시 추가 비용: `len(self.messages)` 조회 1회 (O(1))
- 기존 `_total_chars()` 호출과 AND 조합이므로 실질적 오버헤드 없음

### 4.2 하위 호환성

- `keep_recent` 기본값(4) 변경 없음
- `max_context_chars` 계산 로직 변경 없음
- `__init__` 시그니처 변경 없음 — 새 파라미터 추가 불필요

### 4.3 코드 변경 범위

변경 대상은 `ContextManager.add()` 메서드의 **조건문 1줄**이며, 최소 변경으로 구현 가능해야 한다.

## 5. 범위 외

- 메시지 수 기반 독립 압축 게이트 (메시지만 많을 때 별도 전략)
- 토큰 추정기를 문자 수 대신 실제 토큰으로 전환 (별도 이슈)
- `keep_recent` 동적 조정 (모델 크기에 따른 적응)
- `force_compress`의 게이트 조건 변경
